1.6 Application Flow
--------------------------------------------------------------------------------

High-level user journey:

  1. User opens the application in a browser (Vite SPA).
  2. If not authenticated, they land on /login.
  3. User signs in with email/username/loginId + password.
  4. Auth service validates against the mock users collection.
  5. On success, JWT-like mock token and user profile are stored in
     LocalStorage via Zustand persist (key: smart-ca-auth).
  6. User is redirected to the Dashboard (/).
  7. AppLayout renders:
       - Desktop Sidebar (permission-filtered navigation)
       - Mobile drawer navigation
       - Topbar (search trigger, notifications, theme, profile, logout)
       - Main content area (page outlet)
       - Global Command Palette (Ctrl/Cmd + K)
  8. User navigates modules. Each route checks authentication and required
     permission. Unauthorized access redirects to /unauthorized.
  9. Data for pages is fetched through Services â†’ MockDatabase collections.
 10. Create / Update / Delete / Archive / Duplicate actions write immediately
     into LocalStorage-backed collections.
 11. Theme preference persists separately (smart-ca-theme).
 12. Session inactivity may log the user out unless Remember Me is enabled.
 13. Logout clears auth state and returns to /login.

Data initialization flow on first load:

  â€¢ Database seed version is checked (DB_VERSION = 2).
  â€¢ If not seeded or version mismatch, JSON mock files are loaded into
    LocalStorage under keys smart-ca-db:<collection>.
  â€¢ Subsequent visits read from LocalStorage (user mutations survive refresh).

3.14 State Management â€” How Data Flows
--------------------------------------------------------------------------------

Example: Create Client

  1. User clicks Add Client on ClientsPage (permission: clients.create).
  2. EntityFormModal opens with client Zod schema fields.
  3. User submits; page validates duplicates (PAN/GSTIN) via repository.
  4. ClientService.create(...) inserts into MockDatabase clients collection.
  5. Collection persists to LocalStorage key smart-ca-db:clients.
  6. Optional audit log write via SettingsService.logAudit.
  7. Toast success; React Query invalidateQueries(['clients']).
  8. Table re-fetches and shows the new client.
  9. Browser refresh reloads seed-aware DB from LocalStorage; client remains.

Example: Delete Client with relationships

  1. User clicks Delete on a row.
  2. ConfirmDialog opens with warnings from clientRepository.getDeleteImpact
     (counts invoices, payments, documents, companies, tasks, compliance).
  3. User may Archive instead (soft delete) or force remove.
  4. deleteWithUndo snapshots the record, performs delete/archive, shows
     Undo toast for ~10 seconds.
  5. Undo restores the record into the collection.

5.1 Concept

Smart CA treats the browser as a temporary database server.

On first launch (or after DB version bump / reset):

  JSON seed files â†’ CollectionStore memory â†’ LocalStorage

8. AUTHENTICATION
================================================================================

--------------------------------------------------------------------------------
8.1 Login Flow
--------------------------------------------------------------------------------

  1. User opens /login (GuestRoute: if already authenticated â†’ /).
  2. Submits identifier + password (+ optional Remember Me).
  3. AuthService.login searches users collection by email/username/loginId.
  4. Validates password and active status.
  5. Creates mock token (prefix sca_), writes loginHistory, updates lastLogin.
  6. useAuthStore.login stores token, user (password stripped), session.
  7. setAuthToken updates Axios default Authorization header.
  8. Optional themePreference applied.
  9. Navigate to dashboard.

Demo password for generated users: SmartCA@2025

Example accounts shown on login:
  Admin/Super Admin : rajesh.sharma@smartca.in
  Partner           : priya.patel@smartca.in
  CA                : amit.kumar@smartca.in

--------------------------------------------------------------------------------
8.2 Logout Flow
--------------------------------------------------------------------------------

  1. User selects Sign out in Topbar profile menu.
  2. AuthService.logout clears Axios token.
  3. Auth store clears persisted auth state.
  4. Redirect to /login with success toast.

--------------------------------------------------------------------------------
8.3 Remember Me
--------------------------------------------------------------------------------

When enabled, session expiry is extended (AuthService uses longer expiry
window; session timeout hook skips aggressive inactivity logout).
When disabled, inactivity around 30 minutes triggers logout via
useSessionTimeout + checkSession.

--------------------------------------------------------------------------------
8.4 Protected Routes
--------------------------------------------------------------------------------

ProtectedRoute ensures:
  â€¢ Authenticated
  â€¢ Session not expired
  â€¢ Permission allows route (ROUTE_PERMISSIONS)

Otherwise redirects to /login or /unauthorized.

--------------------------------------------------------------------------------
8.5 LocalStorage (Auth)
--------------------------------------------------------------------------------

Key smart-ca-auth holds serialized Zustand auth persist state so refresh
keeps the user signed in until logout/expiry.

--------------------------------------------------------------------------------
8.6 Mock Authentication Notes
--------------------------------------------------------------------------------

  â€¢ Passwords are plain in mock JSON for demo convenience (NOT production
    safe).
  â€¢ No real email for forgot-password; returns success messaging only.
  â€¢ Token is not cryptographically verified by a server.
  â€¢ Suitable for UI/RBAC demonstration and frontend integration contracts.

--------------------------------------------------------------------------------
8.7 Role Based Navigation
--------------------------------------------------------------------------------

Sidebar and mobile nav call getFilteredNavigation(permissions).
Items without required permission are hidden.
Nested compliance children each require their own *.view permission.
